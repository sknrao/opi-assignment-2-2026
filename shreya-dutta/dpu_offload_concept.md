# DPU Offload Concept: From Software OVS to BlueField-3 vDPA Hardware Offload

## 1. Implemented Software Datapath

The implemented assignment stack uses a local Kubernetes node running KubeVirt, Multus, and an OVS-backed secondary network. The VM receives a default Kubernetes interface for management and a secondary interface connected through Multus to an OVS bridge named `br1`.

The software datapath is:

```text
KubeVirt VirtualMachine
  -> virtio-net device inside guest
  -> QEMU / virt-launcher pod
  -> tap or veth-backed secondary interface
  -> Multus delegated OVS CNI attachment
  -> Open vSwitch bridge br1 on the Kubernetes node
  -> Linux kernel datapath / host CPU packet processing
```

In this mode, the OVS bridge and flow-table lookup are handled in software. Packets generated by the CirrOS VM leave the guest through a virtio interface, reach the KubeVirt launcher process, enter the node-side OVS bridge, and are forwarded according to OVS datapath rules. The file `verification_flows.json` captures the observed OVS flow table after ICMP traffic is generated.

## 2. Bottleneck in the Software Model

The software model is simple and suitable for teaching, but it has performance limitations:

- Packet switching consumes host CPU cycles.
- VM I/O exits through QEMU/virtio processing.
- OVS lookup and forwarding occur in the host kernel or userspace datapath.
- Higher packet rates increase CPU overhead and latency.
- Network isolation and switching compete with application workloads on the same CPU.

This model is acceptable for a local lab but not ideal for high-throughput cloud, NFV, edge, and telco workloads.

## 3. Target BlueField-3 Hardware-Offloaded Architecture

On an NVIDIA BlueField-3 DPU, the datapath can be shifted from host software into the DPU/NIC hardware. The host still orchestrates the VM through Kubernetes and KubeVirt, but packet processing moves closer to the NIC.

The conceptual DPU datapath is:

```text
KubeVirt VirtualMachine
  -> virtio-net or vDPA-backed device
  -> VF / SF / vDPA device exposed to the host
  -> representor interface in switchdev mode
  -> BlueField embedded switch / eSwitch
  -> OVS-DOCA or OVS hardware-offloaded datapath
  -> NIC hardware forwarding
```

In this model, the VM can still see a virtio-compatible network device, but the data movement is accelerated through vDPA and the BlueField hardware datapath. The OVS control plane remains familiar, while selected flow rules are pushed into hardware.

## 4. Role of vDPA

vDPA, or virtio data path acceleration, preserves the virtio programming model for the VM while allowing the data path to be executed by hardware. This means:

- The guest can continue using a virtio-net driver.
- The hypervisor can expose a vDPA-backed interface instead of a purely software virtio path.
- Packet movement can bypass portions of the host software stack.
- CPU intervention is reduced for high-rate traffic.

The VM does not need to understand the DPU internals. It still interacts with a virtio-style device, while the host and DPU provide accelerated forwarding.

## 5. Role of switchdev and Representor Interfaces

In a BlueField offload deployment, the NIC embedded switch is typically placed into `switchdev` mode. In switchdev mode, virtual functions, scalable functions, or vDPA endpoints have corresponding representor interfaces. These representors allow OVS to control traffic policy while the actual forwarding can be executed by the embedded switch hardware.

Conceptually:

```text
VM vDPA/VF endpoint <-> representor interface <-> OVS control plane <-> hardware eSwitch
```

The representor acts as the software-visible handle for a hardware endpoint. OVS programs policy using these representors, and eligible flows can be offloaded into hardware.

## 6. Role of OVS-DOCA

OVS-DOCA is the BlueField-oriented OVS acceleration path. It preserves familiar OVS concepts such as bridges, ports, OpenFlow-style rules, and CLI operations, but uses the DOCA framework and NVIDIA hardware acceleration capabilities underneath.

In the current assignment, `ovs-ofctl dump-flows br1` shows software bridge rules. In a BlueField-3 model, OVS rules may still be visible from the OVS control plane, but eligible flows are installed into the DPU/NIC hardware. The important architectural change is that OVS becomes more of a control plane, while the high-throughput forwarding path is handled by hardware.

## 7. Mapping from Assignment Stack to DPU Stack

| Assignment Component | Software Implementation | BlueField-3 Offload Equivalent |
|---|---|---|
| VM interface | virtio-net via KubeVirt/QEMU | virtio-net backed by vDPA |
| Secondary network | Multus + OVS CNI | Multus + SR-IOV/vDPA-aware CNI path |
| Bridge | Host OVS bridge `br1` | OVS bridge backed by OVS-DOCA / hardware offload |
| Forwarding datapath | Host kernel / software OVS | BlueField eSwitch / NIC hardware |
| Flow visibility | `ovs-ofctl dump-flows br1` | OVS flows plus hardware-offload statistics |
| CPU role | Packet processing and control | Mostly control plane; reduced packet processing |
| Performance expectation | Suitable for lab-scale traffic | Lower CPU load, higher throughput, lower latency |

## 8. Expected Configuration Changes on BlueField-3

A production BlueField-3 implementation would require additional steps beyond the local lab:

1. Install BlueField OS, DOCA runtime, NVIDIA OFED or relevant host drivers.
2. Enable switchdev mode for the relevant NIC/eSwitch.
3. Create VF, SF, or vDPA endpoints for VM attachment.
4. Expose representor interfaces to OVS.
5. Run OVS-DOCA or an OVS build configured for hardware offload.
6. Enable OVS hardware offload, for example through OVS external IDs and DOCA-specific configuration.
7. Attach KubeVirt VM interfaces through an SR-IOV/vDPA-capable device-plugin and CNI path.
8. Verify offload using OVS flow dumps, representor counters, DOCA tooling, and hardware statistics.

## 9. Conceptual Verification in the Offloaded Model

The verification procedure changes from simple software-only checking to hybrid software/hardware checking.

Software lab verification:

```text
ping test
ovs-vsctl show
ovs-ofctl dump-flows br1
ovs bridge packet capture
```

BlueField-3 verification:

```text
ping or traffic generator test
OVS flow table inspection
hardware-offload flag/counter inspection
representor statistics
DPU-side OVS/DOCA logs
NIC/eSwitch counters
host CPU utilization comparison
```

The main success criterion is not only that traffic passes, but that eligible flows are installed and forwarded by hardware rather than remaining entirely in the host software datapath.

## 10. Conclusion

The submitted software stack demonstrates the logical foundations of VM orchestration and OVS-based switching in Kubernetes. The BlueField-3 DPU version keeps the same high-level orchestration idea but changes the datapath placement. Instead of the host CPU performing most switching work, vDPA, switchdev representors, the BlueField embedded switch, and OVS-DOCA move high-rate packet forwarding into hardware. This reduces host CPU load and makes the design suitable for high-performance VM networking, edge, NFV, and cloud infrastructure.
